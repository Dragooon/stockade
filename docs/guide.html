Guide

Setup, configuration, and how things work.

Prerequisites

  • Node.js 22+
  • pnpm
  • Docker (for sandboxed agents)
  • An Anthropic API key

Install

git clone https://github.com/Dragooon/stockade.git
cd stockade
pnpm install

Configure

cp config/config.example.yaml config/config.yaml
cp config/proxy.example.yaml config/proxy.yaml

Add your Anthropic API key:

mkdir -p config/secrets
echo "sk-ant-..." > config/secrets/anthropic-api-key
The config/secrets/ directory is gitignored. Secrets never leave your machine.

Run

Simple (no containers)

Set sandboxed: false on your agents in config.yaml, then:

pnpm start:orchestrator

Full stack (with containers + proxy)

# Terminal 1: Start the credential proxy
pnpm start:proxy

# Terminal 2: Start the orchestrator
pnpm start:orchestrator

The orchestrator auto-provisions containers on first message to a sandboxed agent.

Configuration

Agents

Each agent has a model, tools, system prompt, and permissions:

agents:
  main:
    model: claude-sonnet-4-6
    tools: [Bash, Read, Write, Edit, Glob, Grep]
    subagents: [researcher]
    sandboxed: true
    permissions:
      - "deny:Write(/config/**)"
      - "allow:*"

  researcher:
    model: claude-haiku-4-5-20251001
    tools: [Read, WebSearch, WebFetch]
    sandboxed: true
    permissions:
      - "allow:Read"
      - "allow:WebSearch"
      - "allow:WebFetch"

Permission Rules

Rules are evaluated top-to-bottom, first match wins:

permissions:
  - "deny:Write(/config/**)"        # block config writes
  - "deny:Bash(rm *)"               # block rm commands
  - "allow:Bash(git *)"             # allow git
  - "allow:Read"                    # allow all reads
  - "ask:Write"                     # prompt user for other writes

Path prefixes:

  • / — platform root (~/.stockade)
  • ~/ — home directory
  • ./ — agent working directory
  • // — absolute path

RBAC

Users get roles that control which agents and tools they can use:

rbac:
  roles:
    owner:
      permissions: ["agent:*", "tool:*"]
    user:
      permissions: ["agent:main"]
      deny: ["tool:Bash", "tool:Write"]
  users:
    alice:
      roles: [owner]
      identities:
        discord: "123456789"
        terminal: "alice"

Network Policy (proxy.yaml)

Controls what containers can reach. Deny by default:

policy:
  default: deny
  rules:
    - host: "api.anthropic.com"
      action: allow
    - host: "api.github.com"
      method: "GET"
      action: allow
    - host: "api.github.com"
      action: deny

Credential Injection

The proxy injects API keys into outbound requests. Agents never see secrets:

routes:
  - host: "api.anthropic.com"
    credential: anthropic-api-key    # read from config/secrets/
    inject:
      header: x-api-key

  - host: "api.github.com"
    credential: github-token
    inject:
      header: authorization
      format: "Bearer {value}"

Discord

channels:
  discord:
    enabled: true
    token: ${DISCORD_TOKEN}
    bindings:
      - server: "YOUR_SERVER_ID"
        agent: main
        channels: "*"

The agent receives all messages in bound channels. It uses its system prompt to decide when to respond vs stay silent.

How It Works

Message Flow

  1. Message arrives via channel (terminal/Discord)
  2. RBAC checks caller identity and permissions
  3. Router resolves which agent handles the message
  4. If sandboxed: orchestrator ensures container is running, dispatches via HTTP
  5. If local: orchestrator calls Agent SDK directly
  6. Agent can delegate to sub-agents via ask_agent MCP tool
  7. Response flows back through the channel

Container Lifecycle

Shared by default — one container per agent type, reused across sessions. The worker is stateless; each request carries its own session ID.

On first request to a sandboxed agent:

  1. Orchestrator requests a gateway token from the proxy
  2. Container is created with proxy env vars (no secrets)
  3. Health check passes
  4. All subsequent requests for that agent type reuse the same container

Credential Proxy

Containers set HTTP_PROXY pointing to the proxy. All HTTPS traffic is intercepted via MITM with a local CA cert. The proxy:

  1. Checks network policy (host/path/method allowlist)
  2. Strips outgoing auth headers
  3. Injects the correct credential for the matched route
  4. Forwards to upstream
