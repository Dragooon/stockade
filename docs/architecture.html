Architecture

How the pieces fit together.

Overview

User (Terminal / Discord) | v Orchestrator --- RBAC --- Sessions (SQLite) | |-- Permission Engine (allow / deny / ask) | | | '-- Gatekeeper (AI risk review, optional) | |-- Local Agent (in-process, Agent SDK) | '-- Sandboxed Agent | Container (Docker, internal network, no secrets) | Worker (Hono HTTP server, Agent SDK query()) | Credential Proxy (MITM, per-route injection) | Network Policy (deny-by-default allowlist) | Internet

Packages

PackageWhat it does
orchestratorConfig loading, RBAC, session management, message routing, channel adapters (terminal + Discord), agent dispatch, container lifecycle, permission engine, gatekeeper, task scheduler
workerHTTP server (/run + /health) wrapping the Agent SDK's query(). Runs inside containers. Stateless — each request carries model, system prompt, tools, session ID.
proxyMITM HTTP proxy with TLS interception. Route-based credential injection. SSH tunnel with certificate auth. Gateway API for token management. Provider-agnostic secret resolution (file, 1Password, AWS SM, etc).

Message Flow

1. Message arrives via channel (terminal / Discord) 2. RBAC checks caller identity and permissions 3. Router resolves which agent handles the message 4. Permission engine evaluates tool rules: - allow → proceed - deny → block - ask → gatekeeper reviews risk → auto-approve or prompt user 5. If sandboxed: orchestrator ensures container, dispatches POST /run If local: orchestrator calls Agent SDK query() directly 6. Agent can delegate to sub-agents via ask_agent MCP tool (RBAC re-evaluated at each hop with original caller identity) 7. Response flows back through the channel

Sub-Agent Delegation

The orchestrator injects an ask_agent MCP tool into each agent's session at runtime. When an agent calls it:

  1. The orchestrator intercepts the tool call
  2. RBAC is re-checked using the original caller's identity (not the agent's)
  3. The sub-agent is dispatched (local or container) with its own permissions
  4. Result flows back to the parent agent as the tool response

This means a Discord user's permissions apply even when the orchestrator delegates three levels deep: user → main → engineer → researcher.

Container Lifecycle

Shared mode (default)

One container per agent type, reused across all sessions. The worker is stateless; the Agent SDK manages session state via sessionId.

First request for agent "researcher": 1. ContainerManager.ensure("researcher") 2. Provision: request gateway token from proxy 3. docker create (image, network, env vars, volume mounts) 4. docker start 5. Poll GET /health until ok 6. Cache container URL Subsequent requests: 1. ContainerManager.ensure("researcher") → cached URL 2. POST /run directly

Session-isolated mode

Set container.isolation: session for agents that need filesystem isolation between conversations. Each scope gets its own container, torn down on idle timeout.

Credential Proxy

Containers set HTTP_PROXY / HTTPS_PROXY pointing to the proxy. All HTTPS traffic is intercepted via MITM with a dynamically-generated cert signed by a local CA.

Agent HTTPS request: 1. CONNECT api.anthropic.com:443 2. Proxy terminates TLS (dynamic cert from local CA) 3. Reads plaintext request 4. Network policy check (host + path + method) - deny → 403 - allow → continue 5. Strip outgoing auth headers (authorization, x-api-key) 6. Route match → resolve credential from provider 7. Inject credential into headers 8. Forward to upstream 9. Relay response back to agent

The credential provider is a CLI command template. It works with any secret manager:

# File-based (default)
read: "cat config/secrets/{key}"

# 1Password
read: "op read op://{key}"

# AWS Secrets Manager
read: "aws secretsmanager get-secret-value --secret-id {key} --query SecretString --output text"

Permission Engine

Per-agent, first-match-wins rules. Three actions:

ActionBehavior
allowTool invocation proceeds silently
denyTool invocation is blocked, agent sees error
askHuman-in-the-loop approval required

If no rule matches, the default is ask.

Selector syntax

"allow:*"                      # all tools
"deny:Write"                   # specific tool, all invocations
"allow:Bash(git *)"            # tool + command glob
"deny:Write(/config/**)"       # tool + path glob
"ask:Edit(~/secrets/**)"       # tool + home-relative path

Path prefixes

PrefixResolves to

//Absolute path

Symlinks are resolved to canonical paths. .. traversal is normalized. Case-insensitive matching on Windows.

Gatekeeper

Optional AI-powered risk review for ask rules. Before a tool invocation reaches the user for approval, a gatekeeper agent evaluates it:

Tool invocation matches "ask" rule: 1. Gatekeeper agent reviews the invocation 2. Assigns risk level: low / medium / high / critical 3. If risk ≤ auto_approve_risk threshold: → auto-approve + notify channel 4. If risk > threshold: → present to user with risk review attached → user approves or denies

The gatekeeper is itself an agent defined in config — its model and system prompt are fully customizable.
